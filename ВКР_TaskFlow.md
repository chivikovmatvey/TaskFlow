## ВВЕДЕНИЕ

Широкое использование цифровых технологий привело к появлению новых способов организации совместной работы команд. В современных IT-компаниях, маркетинговых агентствах, образовательных проектах и стартапах планирование и распределение задач становится одним из самых сложных и трудоёмких процессов. Поэтому использование автоматизированных систем управления задачами просто необходимо.

Во-первых, автоматизированный учёт задач облегчает планирование работ и распределение ответственности. Во-вторых, использование информационных систем повышает прозрачность процессов и достоверность отчётности перед руководством, что играет очень важную роль в современном проектном менеджменте. В электронных таск-трекерах можно достаточно легко, быстро и точно получить нужную информацию о статусе задачи, сформировать отчёт об эффективности команды и так далее. Кроме того, фиксация всех действий в системе осуществляется быстрее и точнее, чем при работе с таблицами или мессенджерами.

Создание собственной информационной системы для управления задачами актуально, так как готовые SaaS-решения (Trello, Jira, Asana) либо имеют ограниченный бесплатный тариф, либо ограничены в доступе из России, либо не предоставляют возможности кастомизации под конкретные процессы команды. Собственная система позволяет хранить данные на контролируемом сервере, расширять функциональность по необходимости и обеспечивать высокий уровень безопасности.

Целью данной выпускной квалификационной работы является **проектирование и разработка кроссплатформенного веб-приложения «TaskFlow»** для управления задачами в формате Kanban-доски. Информационная система позволит хранить все необходимые данные о пользователях, проектах (досках), задачах, комментариях, исполнителях и сроках, обеспечивать совместную работу в реальном времени и разграничивать права доступа. При помощи системы можно увеличить производительность команды, сократить время на постановку задач и контроль их выполнения, минимизировать потери информации между сотрудниками.

Для реализации проекта были поставлены следующие задачи:

- описание предметной области управления задачами;
- определение функциональных и нефункциональных требований к системе;
- проектирование концептуальной и логической модели данных;
- построение диаграмм бизнес-процессов и UML-диаграмм взаимодействия;
- проектирование физической структуры реляционной базы данных;
- создание макета пользовательского интерфейса;
- разработка SPA-приложения на React и REST API на Node.js + Express;
- реализация механизма реального времени на базе Socket.IO;
- интеграция с внешним сервисом уведомлений (Telegram-бот);
- тестирование и документирование разработанной системы.

### 1.1 Общие требования к системе

Для реализации автоматизированной информационной системы было принято решение использовать стек **React 19 + Node.js (Express) + Microsoft SQL Server 2022**, развёртываемый в Docker-контейнере. Указанный стек является промышленным, кроссплатформенным и широко применяется в коммерческой разработке.

React — открытая JavaScript-библиотека для построения пользовательских интерфейсов в виде дерева компонентов. Node.js + Express обеспечивает построение REST API; для работы с реальным временем используется библиотека Socket.IO. В качестве СУБД выбран Microsoft SQL Server, что обеспечивает поддержку транзакционной обработки, ссылочной целостности и индексирования.

Главные особенности системы задаются (настраиваются) на уровне конфигурации проекта: набор переменных окружения (`.env`), миграции базы данных, конфигурация сборщика Vite и Tailwind. К ним относятся: параметры подключения к СУБД, секрет JWT-токенов, адрес фронтенда для CORS, параметры внешних сервисов (Telegram Bot API).

### 1.2 Назначение и цели информационной системы

Информационная система предназначена для автоматизации процессов:
- ведения проектов (досок) и колонок (этапов работы);
- создания, редактирования и распределения задач между участниками команды;
- общения по задачам через комментарии;
- хранения файлов, прикрепляемых к задачам;
- учёта времени, потраченного на задачу;
- отслеживания статуса задач и дедлайнов;
- сбора статистики по проекту.

**Основными целями ИС являются:**

- создание единой системы для хранения и обработки данных о задачах команды;
- контроль распределения работы между участниками и сроков исполнения;
- учёт изменений и истории по каждой задаче;

### 1.4 Требования к системе

Развитие концепции системы начинается со сбора данных. На данной стадии ведётся исследование, которое необходимо для развития представлений о процессах командной работы и автоматизируемых бизнес-процессах.

#### 1.4.1 Требования к системе в целом

В выпускной квалификационной работе будет разработана информационная система, которая упростит и автоматизирует управление задачами в проектной команде. ИС позволит хранить все необходимые данные о пользователях, проектах, задачах, комментариях и других важных сущностях. Также благодаря системе можно проводить анализ работы команды, определять уровень эффективности, выявлять задачи, выходящие из срока, и разрабатывать планы развития проекта.

При помощи ИС можно увеличить производительность работы команды, сократить время на постановку задач, обмен статусами и др.

Система должна быть функционально понятной и простой в использовании.

Под операционной системой клиента можно применять любые современные ОС: Windows 10/11, macOS, Linux (любые дистрибутивы), так как клиентом является браузер.

**Аппаратное обеспечение.**

### 1.5 Обзор аналогов

**Trello** — облачная Kanban-доска от Atlassian. Trello обладает простым и понятным интерфейсом, поддерживает drag-and-drop задач, метки, чек-листы, прикрепление файлов, комментарии. Бесплатный тариф ограничивает количество досок и интеграций (Power-Ups). С 2022 года имеет проблемы с доступностью на территории РФ; платежи затруднены.

Преимущества Trello:
- простой интерфейс «из коробки»;
- большая база шаблонов досок;
- интеграции с другими сервисами (Slack, Jira, Google Drive);
- мобильные клиенты для iOS/Android;
- широкая известность.

Недостатки:
- слабая аналитика на бесплатном тарифе;
- ограничения по количеству Power-Ups;
- зависимость от внешнего облака — данные хранятся вне контура заказчика;
- ограниченная доступность из РФ.

**Jira** — система управления задачами и трекинга ошибок от Atlassian, ориентированная на разработчиков и Agile-команды. Поддерживает Scrum/Kanban-доски, спринты, эпики, бэклоги, продвинутые отчёты, гибкую настройку workflow. Высокий порог входа.

Преимущества:
- гибкая настройка процессов под любую методологию;
- мощная аналитика и отчётность;
- интеграция с Bitbucket/Confluence/CI;
- поддержка крупных команд (от 100 человек).

Недостатки:
- высокая стоимость;
- сложность освоения для новичков;
- избыточный функционал для небольших команд;
- ограниченная доступность из РФ.

**YouGile** — российский менеджер задач с поддержкой Kanban, чатов внутри задач, диаграмм Ганта. Полностью локализован, поддерживает on-premise развёртывание.

Преимущества:
- доступен из РФ без ограничений;
- встроенный мессенджер;
- on-premise вариант для корпоративного контура;
- бесплатно до 10 пользователей.

Недостатки:
- меньшая экосистема интеграций по сравнению с Trello/Jira;
- закрытый исходный код;
- ограниченная кастомизация.

**Kaiten, Weeek, Yandex.Tracker** — другие российские аналоги с похожим функционалом и ценовой моделью. Все они являются SaaS-сервисами с закрытым исходным кодом.

**Вывод:** разработка собственной информационной системы TaskFlow обоснована, так как позволяет: разместить данные на собственном сервере (контролируемый контур), не зависеть от внешних SaaS-сервисов и санкционных рисков, гибко расширять функциональность под нужды конкретной команды, а также интегрировать систему с внутренними сервисами заказчика (Telegram-бот, корпоративная почта, Active Directory и т.д.).

Для реализации выпускной квалификационной работы был выбран стек **React 19 + Node.js + MS SQL Server**, поскольку он обеспечивает одновременно производительность, кроссплатформенность и поддержку реального времени.

## 2 ТЕХНИЧЕСКИЙ ПРОЕКТ НА РАЗРАБОТКУ ИНФОРМАЦИОННОЙ СИСТЕМЫ

Технический проект системы представляет собой документ, который описывает в деталях все компоненты и функциональные возможности информационной системы, а также требования к аппаратному и программному обеспечению, которые необходимы для её функционирования.

Технический проект включает в себя план создания и внедрения системы, в том числе этапы разработки, тестирования и внедрения, а также описание методов и процедур, используемых в системе для выполнения нужных задач.

Кроме того, в техническом проекте указаны требования к защите данных, а также качественные характеристики системы и показатели её производительности. Все эти детали предназначены для того, чтобы обеспечить понимание того, как будет работать информационная система и как её следует создавать и использовать.

Этот раздел описывает процесс разработки информационной системы, включающий анализ предметной области, проектирование, тестирование и апробацию. По завершении этапа технического проекта создаётся техническая документация. На этапе «рабочая документация» разрабатывается программный продукт и сопровождающая документация. Документация содержит полную информацию для успешного ввода системы в эксплуатацию, поддержания качества и производительности системы.

### 2.1 Анализ предметной области

Предметная область ИС отражает все аспекты деятельности или знаний, которые будут учтены при её создании. Она определяет, какие данные и какие процессы должны быть автоматизированы, а также как актуальная информация будет храниться, обрабатываться и передаваться между пользователями системы. Важно понимать, что предметную область необходимо анализировать с учётом требований пользователей и рынка, а также учитывать возможные изменения в бизнес-процессах или потребностях пользователей в будущем.

Проектным командам необходимо вести учёт текущих задач, контролировать их выполнение по этапам, а также распределять работу между участниками. Без специализированного инструмента эти процессы либо разрозненны (Excel, мессенджер, почта), либо ведутся в дорогих и закрытых системах.

Работу типичной проектной команды можно разделить на несколько процессов:
- регистрация новой задачи (требования, сроки, исполнитель);
- декомпозиция крупной задачи на подзадачи (чек-лист);
- категоризация задачи метками (баг, фича, документация);
- перевод задачи между этапами (To-Do → In Progress → Review → Done);
- обсуждение деталей задачи с командой (комментарии);
- прикрепление к задаче связанных файлов (макеты, скриншоты, ТЗ);
- учёт времени, потраченного на задачу;
- закрытие задачи и архивирование;
- получение аналитики по проекту (количество задач, распределение по приоритетам, средняя длительность).

#### 2.1.1 Анализ организационной структуры

Целевая организационная структура — **проектная команда** или **отдел разработки** в IT-компании. Типовая структура (рисунок 6):

- **owner** (владелец) — соответствует Project Manager или CTO;
- **admin** (администратор) — соответствует тимлиду;
- **member** (участник) — соответствует исполнителю задачи (разработчик, дизайнер, тестировщик).

#### 2.1.2 Описание бизнес-процессов и потоков данных

На каждом предприятии работа определяется профессиональными отношениями сотрудников, не зависимо от сферы деятельности.

Бизнес-процесс отражает способ выполнения определённой бизнес-деятельности, включая последовательность действий, работников и ресурсов, которые необходимы для её выполнения. Бизнес-процессы могут быть автоматизированы с помощью информационных систем для повышения эффективности и улучшения качества работы команды. ## 3 ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

Главной задачей проекта является обеспечение соответствия системы требуемой функциональности и адаптации к условиям эксплуатации, достижение требуемой пропускной способности, быстрое время реакции на запросы, надёжную работу и безопасность системы, а также обеспечение простоты эксплуатации и поддержки. Эффективность системы напрямую зависит от её производительности, которая сильно зависит от качества проектного решения.

### 3.1 Проектирование структуры базы данных

База данных представляет собой набор связанных между собой данных, относящихся к определённой области. Один из основных подходов к проектированию базы данных — это использование семантического моделирования, которое учитывает значение хранимой в базе данных информации. Для этого используются различные типы ER-диаграмм, которые графически отображают связи и атрибуты объектов в предметной области. Это методология «сущность-связь».

В процессе разработки были выделены следующие информационные объекты:

- пользователи (users);
- доски (boards);
- участники доски (board_members);
- разделы досок (sections);
- участники разделов (section_members);
- колонки (columns);
- задачи (tasks);
- комментарии (comments);
- метки (labels);
- связь задач и меток (task_labels);
- элементы чек-листов (checklist_items);
- прикреплённые файлы (task_attachments);
- учёт времени (time_tracking);
- журнал действий (activity_log).

#### 3.1.1 Описание логической структуры БД

После создания концептуальной модели базы данных следующим шагом является её логическое проектирование. Для реляционной модели существуют строгие правила, которые позволяют превратить ER-диаграмму в логическую структуру данных.

На изображениях представлена именно такая структура для данной базы данных.

**Пользователи (users)** — объект, представляющий зарегистрированного пользователя системы.
Характеризуется: уникальным идентификатором (UUID, обязательный атрибут), e-mail, хешем пароля, полным именем, датами создания и обновления.

**Доски (boards)** — объект, представляющий проект или область работы.
Характеризуется: уникальным идентификатором (UUID), названием, описанием, цветом фона, идентификатором владельца (FK на users), датами создания и обновления.

**Участники доски (board_members)** — объект, реализующий связь many-to-many между пользователями и досками с указанием роли.
Характеризуется: уникальным идентификатором, идентификатором доски (FK), идентификатором пользователя (FK), e-mail приглашения, ролью (`owner`/`admin`/`member`/`viewer`), датой создания. Существует уникальное ограничение `(board_id, user_id)`.

**Разделы (sections)** — объект, представляющий группу досок в Dashboard.
Характеризуется: уникальным идентификатором, названием, идентификатором владельца, цветом, датами создания и обновления.

**Колонки (columns)** — объект, представляющий этап работы (To-Do, In Progress, Done).
Характеризуется: уникальным идентификатором, идентификатором доски (FK), названием, позицией (для сортировки), датой создания.

**Задачи (tasks)** — основной объект системы.
Характеризуется: уникальным идентификатором, идентификатором доски (FK), идентификатором колонки (FK), названием, описанием, позицией внутри колонки, приоритетом (`low`/`medium`/`high`), датой дедлайна, идентификатором исполнителя (FK на users), идентификатором автора (FK на users), флагом архивирования, датой архивирования, датами создания и обновления.

**Комментарии (comments)** — объект, представляющий обсуждение задачи.
Характеризуется: уникальным идентификатором, идентификатором задачи (FK), идентификатором автора (FK на users), текстом, датами создания и обновления.

**Метки (labels)** — объект, представляющий тег задачи.
Характеризуется: уникальным идентификатором, идентификатором доски (FK), названием, цветом, датой создания.

**Связь задач и меток (task_labels)** — объект many-to-many между задачами и метками.
Характеризуется: идентификатором, идентификатором задачи (FK), идентификатором метки (FK). Существует уникальное ограничение `(task_id, label_id)`.

**Элементы чек-листа (checklist_items)** — объект, представляющий пункт чек-листа задачи.
Характеризуется: идентификатором, идентификатором задачи (FK), названием пункта, позицией, флагом выполнения, датой создания.

**Прикреплённые файлы (task_attachments)** — объект, представляющий файл, прикреплённый к задаче.
Характеризуется: идентификатором, идентификатором задачи (FK), именем файла, физическим путём в файловой системе, размером в байтах, MIME-типом, идентификатором загрузившего пользователя (FK), датой создания.

**Учёт времени (time_tracking)** — объект, представляющий сессию работы над задачей.
Характеризуется: идентификатором, идентификатором задачи (FK), идентификатором пользователя (FK), временем начала, временем окончания, длительностью в секундах, заметкой, датами создания и обновления.

**Журнал действий (activity_log)** — объект, фиксирующий действия пользователей.
Характеризуется: идентификатором, идентификатором доски (FK), идентификатором пользователя (FK), типом действия, идентификатором сущности, метаданными в JSON, датой создания.

**Требования целостности:**

- С объектом `boards` должны быть связаны: `columns`, `board_members`, `labels`, `tasks`, `activity_log`. При удалении доски все связанные сущности удаляются каскадно.
- С объектом `users` могут быть связаны: `boards` (как владелец), `board_members`, `tasks` (как исполнитель и/или автор), `comments`, `task_attachments`, `time_tracking`.
- Объект `tasks` обязательно связан с одной `board` и одной `column`. С объектом `tasks` могут быть связаны: `comments`, `task_labels`, `checklist_items`, `task_attachments`, `time_tracking`.
- С объектом `labels` через `task_labels` связан объект `tasks`. Связь many-to-many.
- С объектом `comments` обязательно связан автор (пользователь) и задача.
- С объектом `time_tracking` обязательно связан пользователь и задача.

### 3.3 Проектирование алгоритмов функций информационной системы

Для работы информационной системы необходимо выполнить следующие этапы: получение информации из внутренних или внешних источников (СУБД, файловая система, внешние API), её обработка и представление в удобном формате, а также вывод для отображения в браузере или экспорта.

Интерфейс пользователя содержит компоненты React, кнопки и формы для ввода данных. Для генерации задачи используется выбор нужной информации пользователем (название, колонка, приоритет), после чего система отправляет POST-запрос в REST API, сервер собирает данные и сохраняет их в SQL Server, после чего рассылает событие через Socket.IO всем подключённым клиентам.

Важными функциями являются авторизация и регистрация новых пользователей, которые выполняются через эндпойнты `/api/auth/register` и `/api/auth/login`.

**Основные алгоритмы:**

1. **Алгоритм авторизации:** валидация входных данных → запрос к таблице `users` по e-mail → сравнение хеша пароля через `bcrypt.compare()` → генерация JWT-токена → возврат клиенту.
2. **Алгоритм создания задачи:** валидация прав пользователя на доске → INSERT в таблицу `tasks` → запись в `activity_log` → broadcast события `task:created` в Socket.IO → возврат созданной задачи клиенту.
3. **Алгоритм перемещения задачи (DnD):** обработка события `onDragEnd` от `@dnd-kit` → расчёт новых позиций задач → batch-UPDATE в `tasks` → broadcast события `task:moved` → React Query invalidation на клиентах.
4. **Алгоритм real-time обновлений:** клиент при открытии доски подключается к Socket.IO → подписывается на канал `board:<id>` → при получении события вызывает `queryClient.invalidateQueries(['board', boardId])` (UPDATE-события задач дебаунсятся 500 мс).
5. **Алгоритм отправки Telegram-уведомления:** при назначении задачи / новом комментарии бэкенд проверяет наличие привязанного `chat_id` пользователя → формирует текст сообщения → отправляет через метод `sendMessage` Bot API.

### 3.4 Категории пользователей ИС и их характеристика

Ненадёжное управление доступом пользователей является одной из разновидностей нарушений информационной безопасности. Механизм управления доступом является важным аспектом безопасности, так как целью данного механизма является предотвращение несанкционированного доступа к системе.

**Были выделены категории пользователей ИС:**

- **Гость** — неавторизованный пользователь, имеет доступ только к страницам `/` (главная), `/login` (вход) и `/register` (регистрация);
- **Пользователь** — авторизованный пользователь, имеет доступ к Dashboard, может создавать собственные доски и работать с приглашёнными;
- **Владелец доски (owner)** — пользователь, создавший доску. Имеет полные права: управление досками, колонками, задачами, участниками, удаление доски;
- **Администратор доски (admin)** — пользователь с расширенными правами: управление колонками, задачами, частью настроек доски (без управления участниками и удаления доски);
- **Участник доски (member)** — обычный участник, может создавать, редактировать, перемещать и удалять задачи, оставлять комментарии;
- **Наблюдатель (viewer)** — пользователь только с правом чтения (опционально).

Иерархия ролей: `owner > admin > member > viewer`. Права проверяются на двух уровнях: на бэкенде (middleware `requireBoardRole`) и на фронтенде (хук `useBoardPermissions`).

### 3.5 Структура информационной системы

Совокупность отдельных частей информационной системы называется её структурой, которая состоит из подсистем — частей системы, выделенных по определённому признаку. Если рассматривать общую структуру информационной системы как совокупность подсистем вне зависимости от области применения, то такие подсистемы называются обеспечивающими.

Одна из таких подсистем — информационное обеспечение — имеет целью своевременно формировать и предоставлять достоверную информацию для принятия управленческих решений. Этот элемент является необходимой составляющей структуры информационной системы для успешного функционирования и предоставления актуальной и точной информации для управленческих процессов (рисунок 18).

### 4.1 Выбор средств разработки и реализации функций ИС

Рассмотрим несколько технологий и средств, которые возможно использовать для разработки информационной системы:

**1. React 19** — открытая JavaScript-библиотека для построения пользовательских интерфейсов в виде дерева переиспользуемых компонентов, разработанная Meta. Использует декларативный подход и виртуальный DOM, что обеспечивает высокую производительность отрисовки. Версия 19 принесла нативную поддержку серверных компонентов, automatic batching, useTransition и другие современные API.

Преимущества:
- огромная экосистема (React Router, Redux, TanStack Query, RHF, dnd-kit и т.д.);
- развитое сообщество и документация;
- кроссплатформенность (Web, React Native);
- гибкость и отсутствие навязанной архитектуры.

Недостатки:
- необходимость самостоятельного выбора инструментов (роутер, state-менеджер);
- порог входа выше, чем у Vue.

**2. Vue 3** — прогрессивный фреймворк для построения UI с одной из самых низких порогов входа. Предлагает Composition API и Single-File Components.

Преимущества:
- низкий порог входа;
- встроенная поддержка реактивности;
- официальные роутер и state-менеджер.

Недостатки:
- меньшая, чем у React, экосистема;
- меньшее количество вакансий на рынке РФ.

**3. Angular** — фреймворк от Google с TypeScript «из коробки» и встроенной поддержкой DI.

Преимущества:
- enterprise-готовность;
- встроенный CLI и инструментарий.

Недостатки:
- высокий порог входа;
- избыточен для небольших проектов;
- большой размер бандла.

**4. Express.js** — минималистичный веб-фреймворк для Node.js. Поддерживает middleware, маршрутизацию, обработку запросов/ответов. Используется как стандарт de-facto для построения REST API на Node.

Преимущества:
- низкий порог входа;
- огромная экосистема middleware;
- высокая производительность;
- активное сообщество.

Недостатки:
- отсутствие «встроенной» структуры (приходится организовывать самостоятельно);
- ручная валидация запросов (требуется express-validator).

**5. NestJS** — фреймворк на TypeScript с архитектурой, вдохновлённой Angular. Поддерживает DI, модули, декораторы.

Преимущества:
- структурированность;
- готовая архитектура для крупных проектов.

Недостатки:
- избыточен для небольших проектов;
- дополнительная сложность из-за DI и декораторов.

**6. Django (Python)** — один из самых популярных фреймворков для веб-разработки на Python. Предоставляет ORM, миграции, админ-панель, систему аутентификации.

Преимущества:
- полный набор «из коробки»;
- высокая скорость прототипирования.

Недостатки:
- разный язык на фронте и бэке (Python vs JS);
- меньшая производительность по сравнению с Node.

Во время физического проектирования необходимо определить схему хранения данных, которая зависит от физической структуры выбранной СУБД. Эта структура должна соответствовать логической структуре БД и обеспечивать эффективное размещение и быстрый доступ к данным. Результаты этапа документируются в форме схемы хранения на языке определения данных СУБД.

В качестве СУБД был выбран **Microsoft SQL Server 2022**, развёртываемый в Docker-контейнере. Преимущества выбора:
- поддержка транзакционной обработки и ACID;
- развитая система типов (UNIQUEIDENTIFIER, DATETIMEOFFSET, NVARCHAR(MAX));
- мощный T-SQL с оконными функциями, CTE, JSON-функциями;
- полноценная поддержка внешних ключей с каскадным удалением;
- бесплатная лицензия Developer Edition для разработки;
- лёгкое развёртывание через официальный Docker-образ `mcr.microsoft.com/mssql/server:2022-latest`.

**Окно Kanban-доски (BoardPage):** содержит шапку доски с её названием, кнопкой возврата, индикатором онлайн-пользователей, поисковой строкой, фильтрами и кнопкой «Добавить колонку». Ниже располагаются колонки с задачами.

Для drag-and-drop используется `DndContext` от `@dnd-kit/core`. Каждая колонка обёрнута в `useSortable` (компонент `SortableColumn`), каждая задача — в `useSortable` (компонент `SortableTaskCard`). При перетаскивании рендерится `DragOverlay` с превью карточки.

После завершения DnD вызывается обработчик `onDragEnd`, который рассчитывает новые позиции и отправляет PATCH-запрос в API. Затем React Query инвалидирует кэш `['board', boardId]`.

**Окно модалки задачи (TaskModal):** открывается по клику на карточку. Содержит поля: название, описание, приоритет, дедлайн, исполнитель, метки (`TaskLabels`), чек-лист (`TaskChecklist`), прикреплённые файлы (`TaskAttachments`), учёт времени (`TaskTimeTracking`), комментарии (`CommentItem`).

**Окно настроек (SettingsPage):** содержит управление профилем пользователя, привязку Telegram-аккаунта (`/start <token>` боту), настройку темы (light/dark).

**Окно Insights (InsightsPage):** содержит аналитику по выбранной доске:
- общее количество задач, выполненных задач, средняя длительность;
- распределение задач по приоритетам (диаграмма);
- распределение задач по исполнителям;
- задачи, вышедшие за дедлайн.

**Темная и светлая темы.** Реализованы через TailwindCSS-класс `dark` на корневом элементе. Состояние темы хранится в Zustand-сторе и `localStorage`. Переключение происходит через компонент `ThemeToggle`.

**Адаптивность.** Используется breakpoint-система Tailwind (`sm:`, `md:`, `lg:`, `xl:`). На мобильных экранах колонки доски прокручиваются горизонтально, шапка сворачивается в выпадающее меню.

### 4.5 Тестирование информационной системы

Для проверки информационной системы используются следующие методы:
- **ручное тестирование UI** в браузерах Chrome, Firefox, Safari, Edge;
- **проверка REST API** через утилиты Postman / curl;
- **проверка SQL-запросов** через Microsoft SQL Server Management Studio / Azure Data Studio;
- **проверка real-time** через одновременное открытие доски в нескольких вкладках/браузерах;
- **встроенный отладчик Node.js** для отладки серверной части;
- **React DevTools** и **TanStack Query DevTools** для отладки клиента.

Также имеется встроенный механизм логирования действий пользователей в таблицу `activity_log`, что позволяет проводить аудит и расследовать инцидент при возникновении ошибок.

В процессе тестирования была проведена проверка следующих сценариев:

1. **Регистрация нового пользователя** — успех/ошибка при дублирующем e-mail / слабом пароле.
2. **Авторизация** — успех/ошибка при неверном пароле / несуществующем e-mail.
3. **Создание доски** — корректное добавление в БД и появление в Dashboard.
4. **Создание/удаление колонок** — пересчёт позиций.
5. **CRUD задач** — все операции работают через API и отражаются в UI.
6. **Drag-and-drop задач** — корректный пересчёт позиций при перемещении внутри одной колонки и между колонками.
7. **Real-time** — одновременное открытие доски в двух вкладках; изменение в одной вкладке мгновенно отображается во второй.
8. **Приглашение участника** — отправка приглашения по e-mail, проверка ролей и прав.
9. **Загрузка файлов** — загрузка/удаление вложений.
10. **Telegram-уведомления** — проверка получения уведомлений в личных сообщениях бота.

# ПРИЛОЖЕНИЕ В
## Требования по охране труда при работе с персональными компьютерами

### Общие требования по охране труда

Общие требования безопасности для работников, использующих персональные компьютеры, установлены в инструкции по охране труда. Для работы с ПК допускаются только работники без медицинских противопоказаний и прошедшие обучение по охране труда с группой допуска не ниже 1 по электробезопасности.

Работа на ПК может быть опасной из-за повышенного уровня электромагнитных излучений, статического электричества и напряжённости поля, яркости света и блеска экранов.

Также возможны риски замыкания цепей через человеческое тело или перегрузок мышц кистей рук при выполнении однообразной работы на клавиатуре или мыши. Концентрация на задачах может вызвать умственное напряжение и стресс.

Для обеспечения комфортных условий работы с ПК необходимо учитывать требования к освещению, контрасту и отражениям на экране. Освещённость в зоне размещения рабочего документа должна быть в пределах 300–500 люкс. Расположение экранов мониторов должно исключать возможные отражения и отблески, а также обеспечивать безопасное расстояние между рабочими столами (не менее 2,0 м) и боковыми поверхностями мониторов (не менее 1,2 м). Для снижения электростатического поля можно использовать защитные фильтры на экранах.

В помещении необходимо поддерживать оптимальный уровень влажности и проводить регулярную вентиляцию для создания комфортного климата. Работникам следует предоставить доступ к первичным средствам пожаротушения и аптечке первой помощи при работе с ПК.

Также нужно обеспечить работникам индивидуальную защиту согласно нормативной документации.

При работе с ПК специалисты обязаны соблюдать:

- регулярные перерывы в работе с ПК для предотвращения усталости глаз и мышц;
- использование эргономичной мебели и оборудования, позволяющего правильно расположить тело;
- настройка яркости экрана на оптимальный уровень для минимизации напряжения глаз;
- установка антибликовых фильтров на экраны компьютеров;
- обеспечение хорошего освещения рабочей зоны;
- регулярное проветривание помещений для поддержания качественного воздуха;
- соблюдение правил безопасности при использовании электрических кабелей;
- регулярное обслуживание компьютеров и периферийных устройств;
- обучение сотрудников правильному использованию ПК и основам информационной безопасности.

### Требования по охране труда перед началом работы

При работе с ПК необходимо:

- убедиться в правильной организации рабочего места;
- рабочее место должно быть хорошо освещено;
- клавиатура и мышь должны быть расположены на оптимальном расстоянии от пользователя;
- экран компьютера должен находиться на уровне глаз;
- перерывы необходимы для снятия напряжения с мышц;
- следить за правильной посадкой: спина прижата к стулу, колени выше бедер, ноги на полу или на подставке, локти образуют 90°;
- делать разминку каждый час.

**Запрещается приступать к работе с ПК:**

- при обнаружении неисправности;
- при размещении ПК ближе 1,2 м друг от друга, при расположении рабочих мест с компьютерами в колонку на расстоянии менее 2 м, при рядом расположении дисплеев экранами друг к другу;
- при работе с ПК запрещается производить протирание влажной или мокрой салфеткой электрооборудования, находящегося под напряжением;
- уборку производить только при отключённом оборудовании;
- запрещено использовать в работе неисправное оборудование, при обнаружении неисправности сообщить об этом руководству.

### Требования охраны труда во время выполнения работы

Во время работы на компьютере следует поддерживать рабочее место в чистоте и порядке, держать открытыми вентиляционные отверстия, не загромождать оборудование посторонними предметами и соблюдать санитарные нормы.

В паузах рекомендуется выполнять упражнения для глаз, шеи, рук и тела. Расстояние от глаз до экрана должно быть не менее 60–70 см.

### Требования охраны труда в аварийных ситуациях

При обнаружении аварийных ситуаций (обрыв проводов питания или повреждение электрооборудования) необходимо немедленно выключить питание и сообщить руководителю организации.

При получении травм или возникновении пожара также требуются соответствующие меры безопасности.

### Требования охраны труда по окончании работы

После окончания работы на компьютере нужно выключить все периферийные устройства и блок питания системного блока, а также привести рабочее место в порядок.

В конце рабочего дня следует отключить все электрические приборы и освещение.

## Руководство пользователя

### Назначение программы

Информационная система TaskFlow предназначена для автоматизации процессов ведения проектов и задач команды, обеспечения совместной работы участников в реальном времени, а также распределения задач между сотрудниками и контроля их исполнения.

### Условия выполнения программы

Если ПК пользователя полностью соответствует минимальным системным требованиям (см. раздел 1.4.1) и в браузере включён JavaScript, то приложение запустится корректно.

Минимальные требования: современный браузер (Chrome 120+, Firefox 120+, Edge 120+, Safari 17+), доступ к серверу TaskFlow по сети.

### Выполнение программы

1. **Открытие приложения.** Открыть в браузере адрес, выданный системным администратором (в локальной dev-среде — `http://localhost:3000`).
2. **Регистрация.** При первом использовании нажать «Регистрация», ввести ФИО, e-mail и пароль (минимум 8 символов), нажать «Зарегистрироваться».
3. **Авторизация.** Ввести зарегистрированные e-mail и пароль на странице `/login`. После успешной авторизации происходит переход в Dashboard.
4. **Создание доски.** В Dashboard нажать «+ Создать доску», ввести название и описание, выбрать цвет, нажать «Создать». В новой доске автоматически создаются три колонки: «To-Do», «In Progress», «Done».
5. **Управление колонками.** Добавление колонки — кнопка «+ Колонка» справа от существующих колонок. Переименование/удаление — через меню «⋮» в шапке колонки.
6. **Создание задачи.** Кнопка «+ Добавить задачу» в нижней части колонки. Открывается форма ввода названия. После создания клик по карточке открывает полную модалку.
7. **Drag-and-drop.** Карточку задачи можно перетащить в другую колонку или внутри колонки для изменения позиции. Аналогично можно переупорядочивать сами колонки.
8. **Редактирование задачи.** Клик по карточке открывает модалку, в которой можно изменить название, описание, приоритет, дедлайн, назначить исполнителя, добавить метки, чек-лист, файлы, оставить комментарий, запустить таймер.
9. **Приглашение участников.** Кнопка «👥 N» в шапке доски открывает список участников. Кнопка «+ Пригласить» — форма ввода e-mail и роли.
10. **Аналитика.** Раздел «Insights» в шапке приложения открывает страницу с аналитикой по выбранной доске.
11. **Telegram.** В разделе «Settings» можно привязать Telegram-аккаунт. Скопированный токен отправляется боту командой `/start <token>`. После привязки уведомления о назначении задач и комментариях приходят в Telegram.
12. **Темная/светлая тема.** Переключается кнопкой «🌗» в шапке.
13. **Выход.** Меню пользователя в правом верхнем углу → «Выйти».

### Сообщения оператору

Все системные сообщения выводятся в виде toast-уведомлений в правой нижней части экрана:
- зелёные — об успешных операциях;
- красные — об ошибках (например, «401 — неверный e-mail или пароль», «403 — недостаточно прав»);
- синие — информационные.

В случае серьёзного сбоя обратитесь к системному администратору и сообщите код ошибки из консоли браузера (открывается клавишей F12, вкладка Console).

Перед установкой и эксплуатацией системы рекомендуется внимательно ознакомиться с настоящим руководством.

### Общие сведения

Информационная система TaskFlow предназначена для автоматизации процессов ведения проектов и задач команды, ведения журнала действий пользователей, а также организации совместной работы и обмена данными.

### Установка и настройка

Для установки информационной системы необходимы:

1. **Сервер с установленными:**
   - Node.js 20+ и npm;
   - Docker Engine 24+ (для контейнера MS SQL Server);
   - Git;
   - сетевой доступ к Telegram Bot API (если требуются уведомления).

2. **Получение исходного кода:**
   ```bash
   git clone <репозиторий> TaskFlow
   cd TaskFlow
   ```

3. **Запуск контейнера БД:**
   ```bash
   docker run -e 'ACCEPT_EULA=Y' -e 'MSSQL_SA_PASSWORD=Pass1234!' \
     -p 1433:1433 --name sqlserver \
     -v sqlserver-data:/var/opt/mssql \
     -d mcr.microsoft.com/mssql/server:2022-latest
   ```

4. **Применение DDL-схемы:**
   ```bash
   docker exec -i sqlserver /opt/mssql-tools18/bin/sqlcmd \
     -S localhost -U sa -P 'Pass1234!' -C -i /sql/schema.sql
   ```

5. **Конфигурация переменных окружения backend (`taskflow-backend/.env`):**
   ```
   PORT=5000
   NODE_ENV=production
   DB_SERVER=localhost
   DB_PORT=1433
   DB_NAME=taskflow
   DB_USER=sa
   DB_PASSWORD=Pass1234!
   JWT_SECRET=<сильный_секрет>
   JWT_EXPIRE=7d
   FRONTEND_URL=http://your-domain
   UPLOAD_DIR=./uploads
   TELEGRAM_BOT_TOKEN=<токен_бота>
   TELEGRAM_BOT_USERNAME=TaskFlowBot
   ```

6. **Конфигурация переменных окружения frontend (`taskflow-frontend/.env`):**
   ```
   VITE_API_URL=https://your-domain/api
   ```

7. **Установка зависимостей и запуск:**
   ```bash
   ./start.sh   # для dev
   # или вручную:
   npm install --prefix taskflow-backend && npm start --prefix taskflow-backend &
   npm install --prefix taskflow-frontend && npm run build --prefix taskflow-frontend
   ```

8. **В продакшене** рекомендуется обернуть Node.js-процесс в systemd / pm2, а собранные файлы фронтенда раздавать через nginx с настроенным HTTPS.

### Управление учётными записями

К управлению учётными записями пользователей информационной системы относятся:

- Создание учётных записей (через форму регистрации /register, либо напрямую в таблице `users` через SQL);
- Назначение ролей пользователям в рамках доски (через UI владельца доски, либо через прямое обновление таблицы `board_members`);
- Сброс паролей (через прямое обновление поля `password_hash` в таблице `users` с использованием bcrypt-хеша);
- Удаление пользователей (через DELETE в таблице `users`; каскадно удаляются связанные записи).

### Резервное копирование

Рекомендуется выполнять ежедневное резервное копирование базы данных:

```bash
docker exec sqlserver /opt/mssql-tools18/bin/sqlcmd \
  -S localhost -U sa -P 'Pass1234!' -C \
  -Q "BACKUP DATABASE [taskflow] TO DISK = '/var/opt/mssql/backup/taskflow.bak' WITH INIT"
```

И копировать каталог `taskflow-backend/uploads/` (загруженные файлы).

### Мониторинг

- Логи backend: `journalctl -u taskflow-backend` (если запущен через systemd) либо `pm2 logs`;
- Логи nginx: `/var/log/nginx/access.log` / `error.log`;
- Журнал действий пользователей: таблица `activity_log`.

## Руководство системного программиста

Для составления руководства системного программиста необходимо использовать ГОСТ 19.503-79 ЕСПД, который содержит требования к оформлению и содержанию документа. Руководство должно включать разделы общей информации о программе, настройке, дополнительных возможностях и сообщениях для системного программиста. В зависимости от конкретной ситуации можно объединять или добавлять новые разделы.

### Общие сведения о программе

В общих сведениях о программе следует указать:

- **Языки программирования:** JavaScript (ES2022+), JSX (для React-компонентов), SQL (T-SQL для MS SQL Server), Bash (вспомогательные скрипты).
- **Платформы:** Node.js 20+, браузеры с поддержкой ES2022 (Chrome 120+, Firefox 120+, Safari 17+, Edge 120+).
- **Фреймворки и библиотеки frontend:** React 19, React Router v7, TanStack React Query 5, Zustand 5, TailwindCSS 3, @dnd-kit, Axios, react-hot-toast, socket.io-client.
- **Фреймворки и библиотеки backend:** Express 4, mssql 12 (через tedious), Socket.IO 4, jsonwebtoken, bcryptjs, multer, express-validator, node-telegram-bot-api.
- **СУБД:** Microsoft SQL Server 2022 (Developer Edition), развёртываемая в Docker.
- **Сборщик:** Vite 7 (frontend); Node.js нативно (backend, ES Modules).

Структура программы представлена набором ES-модулей JavaScript, организованных по принципу разделения по слоям (роутер → контроллер → доступ к данным) и по предметным областям (auth, boards, tasks и т.д.).

### Настройка программы

Настройка производится через переменные окружения в файлах `.env` (см. Приложение Д) и через конфигурационные файлы:

- `taskflow-backend/src/db.js` — пул соединений с MS SQL Server, параметры TLS;
- `taskflow-backend/src/realtime.js` — конфигурация Socket.IO (CORS, transports);
- `taskflow-frontend/vite.config.js` — конфигурация Vite (порт, прокси на API);
- `taskflow-frontend/tailwind.config.js` — расширение темы, цвета, плагины;
- `taskflow-frontend/src/services/apiClient.js` — базовый URL Axios, интерсепторы.

При обновлении схемы базы данных необходимо добавить новый файл миграции в `taskflow-backend/sql/migration_*.sql` и применить его через `sqlcmd`. Существующие миграции:
- `migration_sections.sql` — добавление таблиц `sections` и `section_members`;
- `migration_telegram.sql` — добавление полей `telegram_chat_id` и `telegram_link_token` в таблицу `users`.

### Сообщения системному программисту

Сообщения выводятся в стандартный поток `stdout` процесса Node.js. Уровни:
- `console.log` — информационные;
- `console.warn` — предупреждения;
- `console.error` — ошибки.

При необходимости их можно перенаправить в файл логов или в системный journald.

В случае ошибки на стороне БД (например, `ConnectionError: socket hang up`) следует проверить:
- запущен ли Docker-контейнер `sqlserver`;
- корректность пароля в `.env`;
- наличие в `db.js` параметра `cryptoCredentialsDetails: { ciphers: 'DEFAULT@SECLEVEL=0' }` (требуется на Node 20+ из-за изменений OpenSSL 3).

В случае ошибки на стороне фронта (например, `CORS preflight blocked`) проверить совпадение `FRONTEND_URL` в backend `.env` с реальным адресом, с которого открывается приложение.
